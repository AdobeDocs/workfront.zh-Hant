# 建立新文章之前尋找候選文章

當Courtney引進新內容以納入時 — 錄取檔案、客戶或專案經理意見回饋、Slack/成績單貼上、包含附註的熒幕擷圖、支援問題 — **不要建議先建立新文章。** 搜尋存放庫並呈現內容可能適合的現有候選文章。

## 必要的工作流程

1. **請先閱讀新內容**，以識別主題、對象（管理員與一般使用者），以及新增的特定事實/步驟。
2. **使用grep搜尋`help/`候選者**&#x200B;並尋找。 尋找已涵蓋相同功能、區域或工作流程的文章。
3. **傳回候選文章的排名清單** （通常為3-7），每篇都包含：
   - 檔案路徑（可點按的反勾路徑）。
   - 單行理由表示它是候選項（會插入哪個區段，或是為什麼它是最接近的主題比對）。
   - 任何警告訊息（例如「對象不符 — 這是一般使用者，輸入是面向管理員」）。
4. 在起草或編輯之前&#x200B;**詢問Courtney放置位置**。 明確地表達問題，例如，「我應該草稿到哪一個？」 或「希望我把它放入#2中，或者您認為它需要自己的文章嗎？」
5. **如果現有的文章都不適合的話，請只建議新文章**，並解釋為什麼沒有候選者可以運作。

即使內容「明顯」屬於已知文章，仍會提供2-3替代方案，讓Courtney可以確認刊登。 如此一來，內容可能會近乎重複，並部分涵蓋某些位置。

&#x200B;# 編輯說明文章之前先提出變更

當即將編輯`help/`下的任何`.md`說明文章時，**尚未編輯檔案**。 首先，顯示您計畫變更的專案並等待明確核准。

針對您想要接觸的每個檔案：

1. 為檔案命名（路徑）。
2. 將提議的變更顯示為包圍的差異/片段（舊文字和新文字），周圍的內容足夠明確。
3. 簡要說明原因（1個句子）。
4. 以明確要求結束，例如「套用這些變更？」 或「要我繼續嗎？」

只有在Courtney說「是」（或「執行」、「套用」、「執行」等）之後， 您應呼叫編輯工具嗎？

這適用於說明`.md`文章上的&#x200B;**每**&#x200B;次編輯 — 錯字、連結修正、單字交換、術語清理、新區段和重新寫入。 除非Courtney同時明確指出「只要修正它」或「不要詢問，只要編輯」，否則沒有例外。

對於跨越多篇文章的變更：將提議的差異依檔案分組到一個回覆下。 如果集很大，請先列出受影響的檔案，每個檔案各有1行的摘要，然後以批次顯示差異，並在每個批次之前確認。

這會&#x200B;**不**&#x200B;封鎖唯讀研究(grep、read)或在聊天中起草/探索（無檔案寫入）。

&#x200B;# Git認可訊息

起草或產生Git認可訊息時，請遵循此格式。

## 主旨列

- 大約&#x200B;**50個字元或更少**。
- 總結&#x200B;**命令式情境**&#x200B;中的變更（例如「新增……」、「修正……」、「重構……」）。

## 正文

- 在主旨之後於內文前保留一行空白。
- 在大約&#x200B;**72個字元**&#x200B;處換行。
- 使用&#x200B;**專案符號行**&#x200B;來取得說明。 每個專案符號都必須以下列任一專案開頭：
   - **📖** — 當變更&#x200B;**新增**&#x200B;新內容時使用：新檔案、新區段、新功能、新標題、全新行或其他新欄位內容。
   - **✏️** — 當變更&#x200B;**修改**&#x200B;現有工作時使用：編輯現有行、更新現有區段、重構或變更目前內容。

範例：

```
Add refresh token rotation to auth flow

- 📖 Add refresh_tokens table and Alembic migration for schema v3.
- ✏️ Update session middleware to rotate secrets and revoke old tokens.
```

&#x200B;# 提取請求(PR)

## 衍生Jira問題

- 從&#x200B;**目前的Git分支名稱**&#x200B;衍生Jira問題ID （例如，符合`FFENT-8796`的區段）。
- **如果分支名稱包含Jira ID：**&#x200B;請在PR標題中使用該ID，並在`# Context`→`## Jira`中將其連結。
- **如果分支名稱不包含Jira ID：**&#x200B;請從PR標題中省略Jira金鑰。 仍包含`## Jira`，並完全包含： `No ticket`。

## PR標題

**具有Jira ID：** `{type}/{JIRA-ID}- {short task description}`
範例： `feat/FFENT-8001- Add validation for numVariations in OCAPI request`

**不含Jira ID：** `{type}- {short task description}`
範例： `docs- Refresh Object Composite API changelog`

### 認可型別

- **feat** — 新增功能
- **修正** — 修正錯誤
- **重新調整** — 重新寫入/重新建構程式碼而不變更行為
- **perf** — 改善效能
- **style** — 僅格式設定/空白字元；無意義
- **測試** — 新增或修正測試
- **檔案** — 僅限檔案，新增了新內容
- **組建** — 組建工具、CI、相依性、專案版本
- **ops** — 基礎架構，部署，備份，復原
- **雜項** — 雜項（例如`.gitignore`）

## PR內文 — 必要區段

### `# Summary`
變更內容與原因的簡短概觀。

### `# Changes`
依檔案組織。 對於每個接觸的檔案，使用帶有反引號路徑的層級2標題，然後使用專案符號。

```markdown
# Changes

## `path/to/file.ext`
* Concise bullet describing the change in that file.
```

### `# Context`
一律包含`## Jira`子區段。

```markdown
# Context

## Jira
[FFENT-8796](https://jira.corp.adobe.com/browse/FFENT-8796)
```

或者如果沒有票證： `No ticket`
