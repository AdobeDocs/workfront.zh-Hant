# 針對發行版本進行更新(Workfront)

此技能會逐步說明如何為即將推出的功能版本更新Workfront說明文章。 工作流程與`remove-preview-highlighting`相反：新行為正在新增到文章、標籤為預覽，並（稍後在GA）由該其他技能清除。

## 範圍

當&#x200B;**all**&#x200B;為true時套用：

1. 使用者正在更新隨附功能的Workfront說明文章（通常先預覽）。
2. 此變更會引入新行為或UI，而非GA清理。 若要進行GA清理，請改用&#x200B;**remove-preview-highlighting**。
3. 檔案&#x200B;**不是**&#x200B;發行說明。 如需發行說明，請改用&#x200B;**發行說明格式化程式**。
4. 使用者已提供功能內容：至少要提供簡短說明和熒幕擷圖，最好是PRD URL (Adobe Wiki)。

如果範圍不清楚，請先確認，然後再開始。

## 必要的工作流程（人在回圈中）

請&#x200B;**不**&#x200B;大量編輯存放庫。 一次移動一篇文章。 每篇文章後，詢問是否繼續下一篇文章。

### &#x200B;1. 收集特徵內容

向使用者確認：

- **變更的內容** （新行為或UI的1-2句摘要）。
- 新使用者介面的&#x200B;**熒幕擷圖**。 如果提供，請以描述性的kebab大小寫檔案名稱（例如`add-custom-message.png`）儲存在目標文章的`assets/`資料夾下。 如果未提供，請詢問是否要等待一個或繼續預留位置參考。
- **PRD URL** (Adobe Wiki) （如果有的話）。 使用`user-Adobe Wiki Confluence` MCP工具`get_wiki_content`擷取它。 閱讀本檔案以瞭解使用者在UI中看不到之行為：通知副作用、稍後編輯或新增內容時會發生什麼情況、未顯示字元限制、許可權等。
- **可用性**：僅預覽、預覽+快速發行或已正式發行。 這會推動在步驟3中選擇程式碼片段。
- **明確的排除**：使用者要略過的任何文章（例如「範本中不包含此功能」）。

### &#x200B;2. 詳細目錄受影響的文章

使用功能區域的關鍵字（例如`approval workflow`、`document approval`、特定欄位標籤）搜尋存放庫。 建立候選清單：

- 相關`help/quicksilver/.../`樹狀結構中的作法文章。
- 提及功能區域的概述文章和常見問題。
- **排除** `product-announcements/` （發行說明使用不同的技能）。
- **排除**&#x200B;目錄/索引頁面，其中唯一提及的是其他文章的連結文字。
- **排除**&#x200B;使用者在步驟1中說要略過的文章。

向使用者呈現候選者清單。 詢問要更新以及要略過哪一個。 如果同層級文章似乎遺失，則交叉參照`help/quicksilver/TOC.md`。

### &#x200B;3. 挑選預覽程式碼片段

讀取`help/_includes/snippets.md`並按可用性選擇：

| 可用性 | 程式碼片段 |
|---|---|
| 僅限預覽 — 反白顯示的內容是其他正式發行文章中的新內容 | `{{highlighted-preview}}` |
| 僅預覽 — 整篇文章是新的 | `{{highlighted-preview-article-level}}` |
| 預覽+快速發行客戶，一般 | `{{preview-fast-release-general}}` |

如果本季已存在發行版本專屬的程式碼片段，請優先使用該程式碼片段，而非一般程式碼片段。 在套用之前與使用者確認選擇。

### &#x200B;4. 每篇文章 — 先顯示，確定後編輯

對於使用者確認清單中的每篇文章：

1. **讀取檔案。**

2. **決定反白顯示模式。** 詢問適合本文的使用者（答案可能因文章而異）：

   - **每個區段重複**：將`in Production`附加至現有的區段標題。 新增已附加`in Preview`且包裝在`<div class="preview"> ... </div>`中的新區段。 當新行為有意義地變更程式時使用 — 額外的步驟、新影像、新表格列或不同的用語。 操作程式的典型用法。
   - **每行包裝**：在現有區段中新增內嵌的句子，包裝在`<span class="preview"> ... </span>`中。 當新增的句子或兩個自然適合現有段落、表格儲存格或常見問答的句子時使用。
   - **混合**：相同文章中的某些區段會使用每區段重複，其他區段則使用每行包裝。 當文章同時具有程式區段和FAQ樣式區段時，請顯示此選項。

3. **將程式碼片段**&#x200B;置於H1標題後面緊鄰的一行，並在上方和下方加上空白行。 程式碼片段位於&#x200B;**在**&#x200B;介紹段落之前。

4. **將新詳細資料儲存至「一律包含」與「待稽核」中。** 這是最重要的步驟。

   - **一律包含** （自動套用，不提示）：使用者在與UI互動時無法觀察的隱藏行為。 範例：
      - 副作用（例如「編輯這會將電子郵件重新傳送給所有參與者」）
      - 其他物件或後續事件的行為
      - 必要條件和許可權
      - UI中未顯示限制
      - 使用者只能從PRD、檔案或產品團隊學習的任何內容
   - **向上檢閱** （以多重選取形式呈現給具有`AskQuestion`的使用者）：使用者使用功能時可在熒幕上看到的事實。 範例：
      - UI已顯示的字元計數器（例如`0 / 500`）
      - 欄位的預設展開/收合狀態
      - 可見核取方塊的預設選取狀態
      - 欄位旁的按鈕標籤
      - 內嵌顯示的驗證訊息

   針對每個「待稽核」專案，提供一句理由（「協助新手計畫較長的訊息」，「協助未在後續階段看到訊息的使用者知道要擴大訊息」）。 僅包含使用者挑選的專案。 預設原則為「如果使用者在執行任務時可以在熒幕上看到它，請勿重述」 — 但使用者獲得最終呼叫。

5. **建議編輯。** 顯示文章的前後摘錄（或焦點不同的樣式說明），內容包括：片段位置、標題重新命名、新的預覽內內容及其放置位置、熒幕擷取參照，以及任何內嵌`class="preview"`包裝。

6. **等待明確核准** （「確定」、「套用」、「是」）後再寫入檔案。

7. **驗證。** 寫入後，在檔案上執行`ReadLints`並報告任何問題。 重新讀取已變更的區段以確認結構。

8. **將文章層級的編輯**&#x200B;遞延至&#x200B;**寫作品質**&#x200B;技能。 請勿在此重做語音、大寫、粗體規則或連結模式 — 若要求文句通過，請閱讀`~/.cursor/skills/writing-quality/SKILL.md`。

### &#x200B;5. 每篇文章之後

詢問是否要移至下一篇文章、停止、略過或重新造訪目前的文章。

## 內容規則

### 標題

- 剛好將&#x200B;**`in Production`**&#x200B;附加至保留作為生產端參考的現有區段標題。
- 剛好將&#x200B;**`in Preview`**&#x200B;附加至新章節標題。
- 將標題的其餘部分保持句子大小寫（每`writing-quality`）。

### 預覽包裝函式

- **區段層級**：在`<div class="preview"> ... </div>`中換行。 將開頭和結尾標籤分別放在自己的行上，並在每個標籤的上方和下方加上空白行，讓標籤內的標頭和清單仍會顯示。
- **內嵌（句子層級）**：在現有段落、表格儲存格或常見問答集答案中以`<span class="preview"> ... </span>`換行。
- 絕對不要在`<div class="preview">`內巢狀內嵌`<span class="preview">`。

### 程式碼片段位置

- 程式碼片段會緊接在H1後面，上下都會出現空白行。
- 程式碼片段位於&#x200B;**在**&#x200B;介紹段落、`>[!IMPORTANT]`圖說文字和任何存取需求區塊之前。
- 每篇文章一個程式碼片段。

### 熒幕擷圖

- 將新的熒幕擷取畫面儲存至文章的`assets/`資料夾，並附上描述性的kebab大小寫檔案名稱。
- 從新的預覽中區段參照新的熒幕擷取畫面。 如果生產內區段的熒幕擷取畫面不再準確反映功能，請保留原位 — 在GA之前，它仍代表生產行為。
- 請勿虛構熒幕擷圖檔案名稱；如果尚未提供熒幕擷圖，請詢問使用者。

### 附註和提示

- 每個區段最多一個`>[!NOTE]` （或`>[!TIP]`， `>[!IMPORTANT]`， `>[!WARNING]`）。 如果現有區段已有附註，請將相關的新內容以專案符號清單的形式組合到相同的附註中，而不是進行棧疊。

### 不該做什麼

- 請勿在`product-announcements/`下編輯文章。
- 請勿大量編輯；一次一個文章，每次都有明確核准。
- 請勿在未先向使用者呈現的情況下包含可觀察的UI事實。
- 除非使用者明確指示，否則請勿修改`<!-- ... -->`個HTML註解中的內容。
- 請勿變更`author:`或不相關的frontmatter欄位。

## 在展示編輯內容之前的品質檢查

- 程式碼片段會在H1後面出現一次，且會在其專屬的行上及下方顯示空白行。
- 現有區段標題結尾是`in Production`。
- 新區段標題結尾為`in Preview`，且該區段位於`<div class="preview">`內。
- 內嵌新增在`<span class="preview">`內。
- 已編輯檔案上的`ReadLints`是乾淨的。
- 文章會在兩種狀態下正確閱讀（顯示和隱藏預覽內容）。
