# 寫入品質

使用Workfront檔案風格指南和DQTI原則，檢閱和改進Workfront檔案。

## 工作流程

檢閱或編輯內容時：

1. 讀取目標檔案
2. 套用以下規則 — 直接修正問題或標示問題
3. 保留作者的意圖和技術準確性
4. 如需詳細的術語、標點符號和程式規則，請參閱。claude/commands/_writing-quality-reference.md

## 語音與檢視點

- 使用&#x200B;**第二個使用者** （「您可以……」） 為讀取器定址時
- 使用收縮來發聲說話（不要，不要，他們就是）
- 使用「我們建議」以獲得最佳實務，而不是「建議」或「您應該」
- 為一個使用者群組撰寫時，請參考第三人稱中的其他角色(「您可以檢視……但是，Workfront管理員可以限制……」)
- 絕不使用性別代詞 — 重新建構句子或使用「他們」
- 沒有斜線可供選擇 — 使用「或」（「他/她」而不是「他/她」）

## 清晰度(DQTI)

- 每句一個想法
- 儘可能將句子保留在25字以下
- 以動作或結果開頭，而非背景內容
- 使用最簡單的字詞來傳達含義（「使用」不是「利用」、「開始」不是「啟動」、「關於」不是「相關」）
- 避擴音名（「建立」不是「建立」，「設定」不是「的設定」）
- 使用具體具體的語言 — 避免使用模糊的術語（「數個」、「各種」、「一些」）
- 避免雙負值
- 使用選用的代詞(「that」、「who」)增加句子結構的清晰度

## 具體(DQTI)

- 使用特定範例而非抽象說明
- 在範例中包含實際值
- 為確切的UI元素、欄位和區域命名

## 工作方向(DQTI)

- 專注於使用者可以&#x200B;*做什麼*，而不是系統&#x200B;*是什麼*
- 具有命令式動詞的潛在客戶程式（「建立任務」、「設定通知」）
- 為使用者提供足夠的內容以採取行動，但僅此而已
- 包含詳細說明的連結（「如需詳細資訊，請參閱[文章]。」）

## 大小寫

- **標題**：句子一律區分大小寫（「建立任務」而不是「建立任務」）
- **Workfront辭彙**：只有在直接參照UI元素時才會大寫
   - 直接參考：「完成計畫完成日期欄位」。
   - 間接參考：「每項任務必須有計畫完成日期。」
   - 提示：如果您能將「a」或「the」放在詞語前面，通常應該小寫
- **按鈕**：遵循使用者介面大寫，但全大寫按鈕→句子大小寫除外
- **角色**： UI中角色的「系統管理員」；人員的「Workfront管理員」

## 標題

- 在句子大小寫中使用命令命令（「建立任務」、「設定存取層級」）
- 標題應以任務為基礎 — 說明使用者將完成的任務
- 概述文章結尾為「概述」（「專案概述」）
- 使用清除的子目標將長區段分成多個標題

## 交叉引用和連結

使用這些精確模式：

| 狀況 | 格式 |
|-----------|--------|
| 提供資訊後連結 | 「如需詳細資訊，請參閱[文章名稱]。」 |
| 沒有先前資訊的連結 | 「如需詳細資訊，請參閱[文章名稱]。」 |
| 連結至另一篇文章中的章節 | 「如需詳細資訊，請參閱[文章名稱]中的[章節名稱]。」 |
| 同一篇文章中某章節的連結 | 「如需詳細資訊，請參閱本文中的[章節名稱]。」 |

- 在段落中：與文字內嵌
- 在程式步驟中：在步驟之後的新行上
- 多個連結：使用專案符號清單
- 熒幕位置請使用「右上角」/「左上角」

## 粗體和斜體

- **粗體**&#x200B;可點選動作和僅限程式步驟內的UI元素
- 請勿在程式步驟之外加上粗體UI元素
- 請勿粗體顯示對話方塊名稱、頁面名稱、圖示名稱或步驟以外的功能表名稱
- 針對使用者應輸入的文字使用&#x200B;*斜體* （「在URL方塊中輸入&#x200B;*my.workfront.com*」）
- 絕對不要使用斜體來強調重點 — 請改建構句子

## 附註、提示和警告

謹慎使用 — 過度使用會讓讀者看不見它們。

- 每個區段最多一個附註/提示/警告
- 永不棧疊多個備註
- 將相關附註合併成一個附有專案符號清單的附註
- 請勿使用附註來宣告新功能 — 請改用文章文字

| 類型 | 目的 |
|------|---------|
| `>[!NOTE]` | 不符合段落的資訊；避免挫折；版本設定/相容性 |
| `>[!TIP]` | 提供讓使用者生活更輕鬆的建議 |
| `>[!IMPORTANT]` | 步驟或程式的重要資訊 |
| `>[!WARNING]` | 提醒使用者可能發生資料遺失 |

## 程序

- 使用命令標題（「建立任務」）
- 只在必要時才介紹 — 標題應已足夠
- 偏好的簡介格式：不定式片語+冒號（「建立暫時密碼：」）
- 單一步驟程式仍使用編號清單
- 使用「type」或「select」 — 避免「set」或「specify」等含糊的字詞
- 條件步驟：「（條件式）如果您是多個團隊的成員，請選取……」
- 選用步驟：「（選用）若要新增Emoji，請按一下……」
- 說明系統回應發生前或發生後立即的回應
- 多種方法：先記錄最直接的方法，然後使用「或」

### 程式中的直角括弧

- 使用`>`顯示功能表/索引標籤導覽： 「按一下&#x200B;**電子郵件** > **通知**」
- 以粗體顯示按鈕名稱，而非括弧
- 在方括弧中包含空格
- 請勿在主功能表導覽中使用括弧 — 請使用主功能表片段

## 標點符號快速參考

| 規則 | 範例 |
|------|---------|
| 永遠使用牛津逗號 | 「動機、手段和機會」 |
| 程式中的&quot;then&quot;前須有逗號 | 「按一下[設定]，然後按一下[介面]。」 |
| 日期句子中年後的逗號 | 「2016年1月2日刊……」 |
| 僅含月年無逗號 | 「2016年1月」 |
| 清單前的冒號，之後的小寫 | 「選取下列其中一項：選項A」 |
| 程式標題後面沒有冒號 | 「建立任務」（而非「建立任務：」） |
| 省略長破折號 | 可能的話，重新建構句子 |
| 副檔名中的句號 | 「上傳.pdf檔案」 |

如需完整的標點符號和術語規則，請參閱。claude/commands/_writing-quality-reference.md。

## 完整性(DQTI)

- 包含使用者需要瞭解並採取行動的所有資訊
- 不要包含使用者不需要的資訊
- 永遠提供詳細檔案的「詳細資訊」連結
- 檔案系統預設值：清單中的「（預設）」，或句子中的說明

## 組織(DQTI)

- 含漸進式揭露的邏輯結構（概述→詳細資訊→參考）
- 時間順序頁面先顯示最新內容（例如發行說明）
- 每節一個主題
- 對3個以上的平行專案使用清單
- 使用表格進行結構化比較或欄位說明
- 避免文字牆段落 — 分成可消化的區塊

## 文章型別

| 類型 | 目的 | 標題慣例 |
|------|---------|-----------------|
| 概觀 | 內容、圖表、運作方式 — 無程式 | 結尾為「概覽」 |
| 操作說明 | 具有明確步驟的特定工作 | 命令式動詞 |
| 開始使用 | 新使用者的設定資訊+連結 | 「開始使用……」 |
| 參考 | 表格中的欄位說明 | 描述性名詞片語 |

## 驗證檢查清單

編輯之後，請確認：

- [ ]第二個人在整個（「您」），使用的收縮
- [ 所有標題上的]句子大小寫
- [ ]個Workfront辭彙正確大寫（直接與間接參考）
- [ ]僅在程式步驟中的可點按動作上以粗體顯示
- [ ]交叉引用使用標準格式（「如需詳細資訊，請參閱……」）
- [ ]個未棧疊的備註/提示/警告，每個區段最多一個
- [ ]牛津逗號使用方式一致
- [ ]程式步驟使用特定的動詞（型別、選取、按一下 — 不是「設定」或「指定」）
- [ ]個正確標示的條件/選擇性步驟
- [ ]沒有性別代詞
- [ ]句子清晰、具體且以任務為導向
